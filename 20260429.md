##  三阶段 Token Pruner



```text
ViT 前：
  fps=0.5，减少进入 visual encoder 的视频帧数

ViT 前层：
  ViT block 2 后，根据 pkl tracking object/context prior 做第一次剪枝
  keep_ratio = 0.8

ViT 中后层：
  ViT block 6 后，再根据 pkl tracking object/context prior 做第二次剪枝
  keep_ratio = 0.7

ViT 后 / LLM 前：
  对 visual merger 输出后的 tokens 再做一次 object/context budget
  llm_keep_num = 256
```

剪枝单位仍然是 Qwen3-VL visual merger 对齐的 `2×2 merge group`。这样不会破坏 visual merger 对空间合并结构的要求。本次 tracking 来源为：

```text
tracking_source = pkl
tracking files = testdata/D001_03.pkl, D001_04.pkl, D002_01.pkl, D002_02.pkl
```

说明：脚本里策略名仍沿用了 `--pruning_strategy yolo_object` 和输出目录名 `mid_vit_yolo_object_llm256`，但本次实际 bbox 来源是 `tracking_source=pkl`，不是 YOLO 在线检测。

### 0.1 剪枝依据

pkl tracking 结果中每帧包含 bbox 和 track id，然后 bbox 被映射到 Qwen3-VL 的 merged token grid：

```text
pkl bbox 覆盖区域 -> object token
object 周围膨胀区域 -> context token
其余区域 -> background token
```

ViT 内每个剪枝层的 object-prior 分数为：

```text
importance =
  0.40 * object_context_prior
+ 0.20 * token_motion
+ 0.15 * spatial_contrast
+ 0.15 * hidden_delta
+ 0.10 * global_anchor_bonus
```

含义：

1. `object_context_prior`：tracking bbox 内 token 最高，bbox 周围 context 次之。
2. `token_motion`：同一空间位置跨时间变化越大越重要。
3. `spatial_contrast`：与邻域差异越大，越可能是对象边界、门框、栏杆等。
4. `hidden_delta`：当前 block 与上一层特征差异越大，说明视觉塔正在更新该区域。
5. `global_anchor_bonus`：每帧固定保留 4 个全局 anchor，避免整帧背景被剪空。

LLM 前最后一次剪枝使用 object/context budget：优先保留 object token，其次 context token，同时保留少量 global anchors。该步骤主要减少 LLM 侧输入长度，不降低已经发生的 ViT 计算。

## 3. 分阶段结论

### 3.1 ViT 后压缩

流程是：

```text
视频 -> 完整 ViT 编码 -> 得到 full video_tokens -> 选择 256 个 token -> LLM 生成
```

这个阶段的压缩能够减少 LLM 看到的视觉 token，但不能减少 ViT 的计算。因此视觉编码耗时完全不变，平均仍是 `95.72s`。从总耗时看，`object_context` 平均 `102.34s`，和 full 的 `102.46s` 基本一样。

当前 post-ViT compressed generation 还使用了工程近似：为了让压缩后的 token 序列能进入 Qwen3-VL generation，脚本将 `video_grid_thw` 简化为 `[K, 2, 2]`，并用 sequential 3-channel position id 近似 mRoPE。这可以用于验证语义可压缩性，但不是最终 faithful 的 Qwen3-VL mRoPE 实现。下一步需要保留 selected token 的原始 `(t, y, x)` 时空位置编码。

质量上：

1. `uniform` 最弱。平均目标召回只有 `0.146`，长视频 `D002_01` 中会把异常判断成“没有异常”。
2. `object_only` 目标召回最高，平均 `0.936`，但上下文召回只有 `0.032`，容易丢失门禁区域、车辆、通行边界等场景信息。
3. `object_context` 最平衡，目标召回 `0.762`，上下文召回 `0.171`，回答通常能保留人员、门禁区域和异常行为描述。

结论：ViT 后压缩适合证明“object-context 语义 token 可压缩”，但不能证明端到端加速。更精确的第一阶段结论是：object-only 不够，门禁异常理解需要 object-context token。

### 3.2 ViT 前时间降采样基线

流程是：

```text
降低 fps / 减少输入帧 -> 少量 patch 进入 ViT -> ViT 编码 -> LLM 生成
```

本轮用 `fps=0.5` 代替 `fps=2`。四段视频的视觉编码耗时从平均 `95.72s` 降到 `23.72s`，总耗时从 `102.46s` 降到 `27.95s`，这是唯一显著降低端到端耗时的方案。需要强调的是，这个实验是 pre-ViT temporal downsampling baseline，不是 object-level compression。

分视频看：

| 视频 | fps=2 full 总耗时/s | fps=0.5 总耗时/s | 加速 |
|---|---:|---:|---:|
| D001_03 | 54.44 | 17.09 | 3.19x |
| D001_04 | 55.11 | 17.40 | 3.17x |
| D002_01 | 112.77 | 31.16 | 3.62x |
| D002_02 | 187.54 | 46.13 | 4.07x |

质量上，pre-ViT 的速度收益明显，但有语义损失：

1. `D001_03` 仍能判断存在异常，并能描述人员和门禁区域。
2. `D001_04` 能描述两名男子、铁门、车辆，但异常判断变得保守，没有像 `fps=2 object_context` 那样明确描述快速穿越或绕过门禁。
3. `D002_01` 从 `fps=2 full/object_context` 的“存在翻越或异常进出”变成“没有异常”，说明低 fps 会漏掉关键瞬间。
4. `D002_02` 仍判断无异常，与 full 基本一致。

结论：真正加速必须发生在 ViT 前，但不能简单低 fps；需要 tracking-guided key frame / key region selection 来避免短时异常漏检。

### 3.3 ViT 中间剪枝探索

流程是：

```text
视频 -> patch_embed + 第 1 个 ViT block -> object/context 选择 256 个 merge group -> 后续 ViT block -> LLM 生成
```

该实验确认已经走 FlashAttention 路径。之前报错栈进入了 `transformers/integrations/flash_attention.py` 和 `flash_attn_varlen_func`，说明不是 eager/sdpa。

但是耗时没有明显下降。平均视觉编码耗时 `95.04s`，几乎等于 full 的 `95.72s`；总耗时 `101.90s`，也几乎不变。

可能原因：

1. Qwen3-VL 视觉塔的耗时不只来自后续 block 的 token 数，也包括 patch embedding、位置编码、早期 block、merger 和 FlashAttention 调度。
2. 当前实现是 Python monkey-patch，不能保证底层 kernel 得到理想的稀疏/裁剪收益。
3. 中间剪枝后仍要维护不规则 `cu_seqlens`、position embedding 和 merge group，额外调度可能抵消 token 数减少。
4. 第 1 层后剪枝会过早丢失视觉上下文，导致质量下降。

质量上，mid-ViT 明显不如 ViT 后 `object_context`：

1. `D001_03` 回答变成普通停车场/入口描述，没有明确指出异常行为。
2. `D001_04` 直接判断没有异常，而 ViT 后 `object_context` 能描述疑似快速穿越/绕过门禁。
3. `D002_01` 判断没有异常，而 full 和 ViT 后 `object_context` 都倾向存在异常。
4. `D002_02` 仍判断没有异常，与 full 一致。

结论：当前 Python monkey-patch 版本的 mid-ViT 剪枝不推荐作为主线。它既没有明显加速，也损害了异常识别质量。但这不能说明 mid-ViT 方法本身不可行，失败可能来自 Python patch 开销、FlashAttention 不规则序列调度开销、`cu_seqlens` 重构开销、早期裁剪导致上下文丢失，以及视觉塔耗时不只来自后续 block。

### 3.4 ViT 中间早期轻剪：`early_mild after2 keep70%`

新增实验目录：`early_mild_after2_keep70_fps05_4videos`。

运行配置：

```text
fps = 0.5
pruning_strategy = early_mild
prune_after = 2
keep_ratio = 0.7
anchors_per_frame = 4
max_videos = 4
max_new_tokens = 128
```

该代码的 token 压缩方式如下：

```text
视频帧
↓
Qwen3-VL processor 得到 pixel_values_videos
↓
visual.patch_embed
↓
ViT block 1
↓
ViT block 2
↓
把 raw patch 按 2×2 spatial merge group 聚合成 group 特征
↓
为每个 group 计算 early importance score
↓
保留每帧 global anchors + top 70% group
↓
把保留的 group 展开回 raw patch indices
↓
继续跑后续 ViT blocks 和 visual merger
↓
LLM generation
```

早期轻剪不是用 tracking bbox 直接判断“人/门禁”，而是按低层视觉变化来剪。当前分数为：

```text
importance =
  0.35 * hidden_delta
+ 0.25 * token_motion
+ 0.20 * spatial_contrast
+ 0.10 * hidden_norm
+ 0.10 * global_anchor_bonus
```

其中：

1. `hidden_delta`：block2 与 block1 的 group feature 差异，表示该区域是否被视觉塔明显更新。
2. `token_motion`：相同空间位置跨时间的 feature 变化，表示该区域是否有运动或状态变化。
3. `spatial_contrast`：与上下左右邻域的 feature 差异，表示是否处在边界、对象轮廓、门框/栏杆等区域。
4. `hidden_norm`：特征强度。
5. `global_anchor_bonus`：每帧固定保留 4 个均匀 anchor，避免某些时间片或全局背景完全丢失。

当前实现要点：

1. 剪枝单位是 Qwen3-VL visual merger 对齐的 `2×2 merge group`，不是单个 raw patch。
2. 现在实现的是 `keep/drop`，还没有真正做 token merge；`merge_ratio=0.2` 目前只作为实验记录字段。
3. `cu_seqlens` 会在剪枝后按每帧保留的 raw patch 数重新构造，以适配 FlashAttention 的 variable length 输入。
4. 这仍是 Python monkey-patch 实验版本，不是最终高效 kernel 实现。

## 4. 模型推理文本质量评估

本节关注模型最终生成的文字质量，而不是只看 token 召回或耗时。评价维度包括：

1. 异常判断是否与 `fps=2 full` 基线一致。
2. 是否描述到人员、车辆、门禁区域等关键对象。
3. 是否能解释动作与门禁之间的关系，例如快速穿越、绕过门禁、异常进出。
4. 是否存在明显重复、幻觉或过度保守判断。

### 4.1 异常判断一致性

| 视频 | baseline full `fps=2` | post-ViT `object_context` | pre-ViT `fps=0.5` | mid-ViT `object_context` | mid-ViT `early_mild` |
|---|---|---|---|---|---|
| D001_03 | 判断存在异常 | 判断存在异常 | 判断存在异常 | 主要描述通行，未明确异常 | 判断存在异常 |
| D001_04 | 描述人员在门禁旁反复活动 | 判断存在异常，并描述疑似快速穿越/绕过门禁 | 描述门禁互动，但异常判断偏保守 | 判断无异常 | 判断无异常 |
| D002_01 | 判断存在翻越门禁 | 判断存在异常 | 判断无异常 | 判断无异常 | 判断无异常 |
| D002_02 | 判断无异常 | 判断无异常 | 判断无异常 | 判断无异常 | 判断无异常 |

从异常判断看，`post-ViT object_context` 与 full 基线最接近。`pre-ViT fps=0.5` 虽然速度明显更快，但在 `D002_01` 上漏掉异常。原始 `mid-ViT object_context` 在 `D001_03`、`D001_04`、`D002_01` 上都明显偏保守。新增 `early_mild` 能恢复 `D001_03` 的异常判断，但 `D001_04` 和 `D002_01` 仍然判断无异常。

### 4.2 各方案文本质量

| 方案 | 文本质量表现 |
|---|---|
| baseline full `fps=2` | 信息最完整，能覆盖人员、门禁和异常判断；缺点是重复明显，部分回答会反复描述相同路线 |
| post-ViT `uniform` | 高压缩下容易漏目标，长视频风险最大；`D002_01` 中错误判断无异常 |
| post-ViT `object_only` | 人员保留强，能看到主体动作；但门禁区域和上下文弱，容易把动作描述成孤立的人物移动 |
| post-ViT `object_context` | 当前 ViT 后压缩里文本质量最好，能兼顾人员、门禁区域和异常行为关系 |

### 4.3 代表性现象

`D001_03` 中，full 和 post-ViT `object_context` 都能判断存在异常。post-ViT `object_context` 明确描述“人员从黑色金属门禁区域进入/离开，没有使用门禁系统”，说明目标和门禁上下文仍然保留。pre-ViT `fps=0.5` 也能判断异常，但描述更偏向粗粒度人物移动。原始 mid-ViT 主要描述停车场入口、车辆和人员经过，没有明确异常判断；新增 `early_mild after2 keep70%` 能重新判断存在异常，但回答仍有重复。

`D001_04` 中，post-ViT `object_context` 能描述两名人员在铁门旁、快速跑向铁门、疑似绕过门禁。pre-ViT `fps=0.5` 只描述两名男子与门禁系统互动，异常性变弱。mid-ViT 直接输出“没有看到任何人员翻越、攀爬、绕过门禁、尾随、异常进出等行为”，与 post-ViT 的异常判断相反。

`D002_01` 是最能区分质量的样本。baseline full 判断存在翻越门禁，post-ViT `object_context` 也判断存在异常，并保留门禁小屋、人员、三轮车等场景元素。pre-ViT `fps=0.5`、原始 mid-ViT 和新增 `early_mild` 都判断无异常，说明关键问题不是只靠中间层轻剪就能解决，而是低 fps 已经丢掉了短时关键动作。

`D002_02` 中，full 和各压缩方案基本都判断无异常。这个样本说明在正常视频上压缩不会必然引入误报，但它不能证明异常检测能力。

从这个视频可以看到：`post-ViT object_context` 的文本最适合异常理解展示。它虽然不能加速总流程，但生成质量最接近 full，并且比 full 更聚焦。`pre-ViT fps=0.5` 速度优势明显，但动作细节损失较多。`mid-ViT object_context` 在该视频上没有抓住异常语义。

### 4.5 文本质量结论

综合文本质量和耗时：

1. 质量最稳的是 `post-ViT object_context`，但它不带来端到端加速。
2. 速度最优的是 `pre-ViT fps=0.5`，但会牺牲短时异常识别质量。
3. 当前 `mid-ViT object_context` 既没有明显加速，也降低异常判断质量，不建议作为主线。
4. 新增 `mid-ViT early_mild after2 keep70%` 是一个比 after1 keep256 更合理的早期轻剪原型，但在 `fps=0.5` 下仍漏掉 `D002_01`，说明下一步需要先解决关键帧采样，而不是只调中间层剪枝比例。
5. 后续最有价值的方向是把 `post-ViT object_context` 的目标/上下文选择思想前移到 ViT 输入阶段，用 tracking 引导关键帧或关键区域输入，而不是简单降 fps。

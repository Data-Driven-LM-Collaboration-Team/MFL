# 不同阶段与不同压缩方案对比总结

模型：`Qwen3-VL-2B-Instruct`  
数据：`D001_03`、`D001_04`、`D002_01`、`D002_02` 四段门禁监控视频

## 1. 对比对象

本次对比覆盖三个压缩阶段：

| 阶段 | 实验目录 | 压缩方式 |
|---|---|---|
| 不压缩基线 | `tracking_token_compare_fps2_20260428_165741` | `fps=2`，完整 ViT 编码，完整视觉 token 输入 LLM |
| ViT 后压缩 | `tracking_token_compare_fps2_20260428_165741` | 完整 ViT 编码后，从 `video_tokens` 中选 256 个 token |
| ViT 前压缩 | `pre_vit_fps05_full_20260428` | 将视频输入从 `fps=2` 降到 `fps=0.5`，减少进入 ViT 的帧数和 patch 数 |
| ViT 中间压缩 | `mid_vit_prune_after1_framewise_20260428_v2` | ViT 第 1 个 block 后按 object/context 选择 256 个 merge group，再继续跑后续 ViT block |

ViT 后压缩中重点比较了 `uniform`、`object_only`、`object_context` 三种代表方法。完整实验还包含 `motion`、`object_context_bg25`、`object_context_bg0`，但核心结论主要由这三类方案即可说明。

## 2. 平均耗时与召回

四段视频平均结果如下：

| 方案 | 阶段 | 平均视觉 token 数 | 平均压缩率 | 目标召回 | 上下文召回 | 视觉编码耗时/s | 生成耗时/s | 总耗时/s |
|---|---|---:|---:|---:|---:|---:|---:|---:|
| baseline full `fps=2` | 不压缩 | 2278 | 1.00 | 1.000 | 1.000 | 95.72 | 6.65 | 102.46 |
| post-ViT `uniform` | ViT 后 | 256 | 8.90 | 0.146 | 0.153 | 95.72 | 6.72 | 102.54 |
| post-ViT `object_only` | ViT 后 | 256 | 8.90 | 0.936 | 0.032 | 95.72 | 7.48 | 103.30 |
| post-ViT `object_context` | ViT 后 | 256 | 8.90 | 0.762 | 0.171 | 95.72 | 6.53 | 102.34 |
| pre-ViT `fps=0.5` | ViT 前 | 555 | 约 4.10 | 1.000 | 1.000 | 23.72 | 4.05 | 27.95 |
| mid-ViT `object_context` | ViT 中间 | 256 | 8.90 | 0.762 | 0.173 | 95.04 | 5.39 | 101.90 |

注意：pre-ViT 的表中压缩率是相对 `fps=2 full` 的平均 token 数估算；该实验目录自身的 `compression_ratio=1.0`，因为在 `fps=0.5` 输入下没有再做 token 裁剪。

## 3. 分阶段结论

### 3.1 ViT 后压缩

流程是：

```text
视频 -> 完整 ViT 编码 -> 得到 full video_tokens -> 选择 256 个 token -> LLM 生成
```

这个阶段的压缩能够减少 LLM 看到的视觉 token，但不能减少 ViT 的计算。因此视觉编码耗时完全不变，平均仍是 `95.72s`。从总耗时看，`object_context` 平均 `102.34s`，和 full 的 `102.46s` 基本一样。

质量上：

1. `uniform` 最弱。平均目标召回只有 `0.146`，长视频 `D002_01` 中会把异常判断成“没有异常”。
2. `object_only` 目标召回最高，平均 `0.936`，但上下文召回只有 `0.032`，容易丢失门禁区域、车辆、通行边界等场景信息。
3. `object_context` 最平衡，目标召回 `0.762`，上下文召回 `0.171`，回答通常能保留人员、门禁区域和异常行为描述。

结论：ViT 后压缩适合证明“语义 token 可压缩”，但不能证明端到端加速。

### 3.2 ViT 前压缩

流程是：

```text
降低 fps / 减少输入帧 -> 少量 patch 进入 ViT -> ViT 编码 -> LLM 生成
```

本轮用 `fps=0.5` 代替 `fps=2`。四段视频的视觉编码耗时从平均 `95.72s` 降到 `23.72s`，总耗时从 `102.46s` 降到 `27.95s`，这是唯一显著降低端到端耗时的方案。

分视频看：

| 视频 | fps=2 full 总耗时/s | fps=0.5 总耗时/s | 加速 |
|---|---:|---:|---:|
| D001_03 | 54.44 | 17.09 | 3.19x |
| D001_04 | 55.11 | 17.40 | 3.17x |
| D002_01 | 112.77 | 31.16 | 3.62x |
| D002_02 | 187.54 | 46.13 | 4.07x |

质量上，pre-ViT 的速度收益明显，但有语义损失：

1. `D001_03` 仍能判断存在异常，并能描述人员和门禁区域。
2. `D001_04` 能描述两名男子、铁门、车辆，但异常判断变得保守，没有像 `fps=2 object_context` 那样明确描述快速穿越或绕过门禁。
3. `D002_01` 从 `fps=2 full/object_context` 的“存在翻越或异常进出”变成“没有异常”，说明低 fps 会漏掉关键瞬间。
4. `D002_02` 仍判断无异常，与 full 基本一致。

结论：ViT 前压缩是真正有效的加速方向，但需要解决低 fps 漏检异常的问题。

### 3.3 ViT 中间压缩

流程是：

```text
视频 -> patch_embed + 第 1 个 ViT block -> object/context 选择 256 个 merge group -> 后续 ViT block -> LLM 生成
```

该实验确认已经走 FlashAttention 路径。之前报错栈进入了 `transformers/integrations/flash_attention.py` 和 `flash_attn_varlen_func`，说明不是 eager/sdpa。

但是耗时没有明显下降。平均视觉编码耗时 `95.04s`，几乎等于 full 的 `95.72s`；总耗时 `101.90s`，也几乎不变。

可能原因：

1. Qwen3-VL 视觉塔的耗时不只来自后续 block 的 token 数，也包括 patch embedding、位置编码、早期 block、merger 和 FlashAttention 调度。
2. 当前实现是 Python monkey-patch，不能保证底层 kernel 得到理想的稀疏/裁剪收益。
3. 中间剪枝后仍要维护不规则 `cu_seqlens`、position embedding 和 merge group，额外调度可能抵消 token 数减少。
4. 第 1 层后剪枝会过早丢失视觉上下文，导致质量下降。

质量上，mid-ViT 明显不如 ViT 后 `object_context`：

1. `D001_03` 回答变成普通停车场/入口描述，没有明确指出异常行为。
2. `D001_04` 直接判断没有异常，而 ViT 后 `object_context` 能描述疑似快速穿越/绕过门禁。
3. `D002_01` 判断没有异常，而 full 和 ViT 后 `object_context` 都倾向存在异常。
4. `D002_02` 仍判断没有异常，与 full 一致。

结论：当前 mid-ViT 剪枝实现不推荐作为主线。它既没有明显加速，也损害了异常识别质量。

## 4. 模型推理文本质量评估

本节关注模型最终生成的文字质量，而不是只看 token 召回或耗时。评价维度包括：

1. 异常判断是否与 `fps=2 full` 基线一致。
2. 是否描述到人员、车辆、门禁区域等关键对象。
3. 是否能解释动作与门禁之间的关系，例如快速穿越、绕过门禁、异常进出。
4. 是否存在明显重复、幻觉或过度保守判断。

### 4.1 异常判断一致性

| 视频 | baseline full `fps=2` | post-ViT `object_context` | pre-ViT `fps=0.5` | mid-ViT `object_context` |
|---|---|---|---|---|
| D001_03 | 判断存在异常 | 判断存在异常 | 判断存在异常 | 主要描述通行，未明确异常 |
| D001_04 | 描述人员在门禁旁反复活动 | 判断存在异常，并描述疑似快速穿越/绕过门禁 | 描述门禁互动，但异常判断偏保守 | 判断无异常 |
| D002_01 | 判断存在翻越门禁 | 判断存在异常 | 判断无异常 | 判断无异常 |
| D002_02 | 判断无异常 | 判断无异常 | 判断无异常 | 判断无异常 |

从异常判断看，`post-ViT object_context` 与 full 基线最接近。`pre-ViT fps=0.5` 虽然速度明显更快，但在 `D002_01` 上漏掉异常。`mid-ViT object_context` 在 `D001_03`、`D001_04`、`D002_01` 上都明显偏保守，容易把异常样本说成普通通行或无异常。

### 4.2 各方案文本质量

| 方案 | 文本质量表现 |
|---|---|
| baseline full `fps=2` | 信息最完整，能覆盖人员、门禁和异常判断；缺点是重复明显，部分回答会反复描述相同路线 |
| post-ViT `uniform` | 高压缩下容易漏目标，长视频风险最大；`D002_01` 中错误判断无异常 |
| post-ViT `object_only` | 人员保留强，能看到主体动作；但门禁区域和上下文弱，容易把动作描述成孤立的人物移动 |
| post-ViT `object_context` | 当前 ViT 后压缩里文本质量最好，能兼顾人员、门禁区域和异常行为关系 |
| pre-ViT `fps=0.5` | 文字更短，速度最好；但低帧率会丢失短时关键动作，异常判断更保守 |
| mid-ViT `object_context` | 当前实现质量不稳定，虽然 token 数少，但异常样本容易被说成无异常 |

### 4.3 代表性现象

`D001_03` 中，full 和 post-ViT `object_context` 都能判断存在异常。post-ViT `object_context` 明确描述“人员从黑色金属门禁区域进入/离开，没有使用门禁系统”，说明目标和门禁上下文仍然保留。pre-ViT `fps=0.5` 也能判断异常，但描述更偏向粗粒度人物移动。mid-ViT 则主要描述停车场入口、车辆和人员经过，没有明确异常判断。

`D001_04` 中，post-ViT `object_context` 能描述两名人员在铁门旁、快速跑向铁门、疑似绕过门禁。pre-ViT `fps=0.5` 只描述两名男子与门禁系统互动，异常性变弱。mid-ViT 直接输出“没有看到任何人员翻越、攀爬、绕过门禁、尾随、异常进出等行为”，与 post-ViT 的异常判断相反。

`D002_01` 是最能区分质量的样本。baseline full 判断存在翻越门禁，post-ViT `object_context` 也判断存在异常，并保留门禁小屋、人员、三轮车等场景元素。pre-ViT `fps=0.5` 和 mid-ViT 都判断无异常，说明二者在短时异常或关键动作捕获上存在漏检风险。

`D002_02` 中，full 和各压缩方案基本都判断无异常。这个样本说明在正常视频上压缩不会必然引入误报，但它不能证明异常检测能力。

### 4.4 D001_03.avi 生成内容案例

视频路径：`/home/expand_disk/code_repository/mfl/yong_task/token_pruner_merge/demotest/20260428/testdata/D001_03.avi`

该视频适合作为单例展示，因为它同时包含门禁区域、车辆背景、多名人员移动，以及异常判断需求。不同压缩阶段的生成内容差异如下：

| 方案 | 生成内容摘要 | 质量判断 |
|---|---|---|
| baseline full `fps=2` | 判断“存在人员翻越、攀爬、绕过门禁、尾随、异常进出等行为”；描述白衣男子、黑衣男子从画面左侧进入并经过黑色铁门 | 异常判断明确，主体和门禁都有；但重复严重，多次重复“从门禁区域左侧进入、向右移动、经过黑色铁门” |
| post-ViT `uniform` | 描述停车场/入口区域、车辆、电动车和铁门；描述多人从左右进入并经过铁门 | 场景描述较完整，但异常判断弱，没有明确抓住“绕过/异常进出”的行为关系 |
| post-ViT `object_only` | 描述门禁区域、多名行人和车辆；强调白衣/黑衣男子经过黑色金属门禁 | 人员主体保留较好，但上下文和异常解释不足，容易变成“有人经过门禁”的普通描述 |
| post-ViT `object_context` | 判断存在异常；描述白衣男子、黑衣男子从黑色金属门禁区域进入/离开，且没有使用门禁系统、直接穿过门禁区域 | 该视频上质量最好，既保留人员主体，又保留门禁上下文和异常行为解释 |
| pre-ViT `fps=0.5` | 判断存在异常；描述红色电动车、白色轿车、白衣男子和黑衣男子从左侧走向门禁区域 | 仍能判断异常，速度快；但动作细节更粗，异常解释不如 post-ViT `object_context` 精确 |
| mid-ViT `object_context` | 描述停车场入口、汽车、摩托车、金属栅栏门和人员从左侧进入后向右离开 | 场景元素有保留，但没有明确异常判断，生成内容偏普通通行描述 |

从这个视频可以看到：`post-ViT object_context` 的文本最适合异常理解展示。它虽然不能加速总流程，但生成质量最接近 full，并且比 full 更聚焦。`pre-ViT fps=0.5` 速度优势明显，但动作细节损失较多。`mid-ViT object_context` 在该视频上没有抓住异常语义。

### 4.5 文本质量结论

综合文本质量和耗时：

1. 质量最稳的是 `post-ViT object_context`，但它不带来端到端加速。
2. 速度最优的是 `pre-ViT fps=0.5`，但会牺牲短时异常识别质量。
3. 当前 `mid-ViT object_context` 既没有明显加速，也降低异常判断质量，不建议作为主线。
4. 后续最有价值的方向是把 `post-ViT object_context` 的目标/上下文选择思想前移到 ViT 输入阶段，用 tracking 引导关键帧或关键区域输入，而不是简单降 fps。



## 

当前实验说明：**ViT 后压缩能保持语义但不明显加速，ViT 前压缩能显著加速但可能漏掉短时异常，ViT 中间剪枝在当前实现下加速不明显也不稳定。**
